# /scout — 에이전트 적합지 발굴 + 설계

아무 프로젝트에 들어가서, **"AI가 안전하게 *행동*해도 되는 자리"**(= 사람이 반복하는 운영 판단)를 *프로젝트 이해 + 짧은 인터뷰*로 찾아 **적합도순으로 랭킹**하고, 선택된 후보를 **AI가 그대로 만들 수 있는 설계 스펙**으로 떨궈준다.

박제하는 것은 하나의 설계 사고다 — *생성/검증 분리 · 승인 게이트(HITL) · 운영을 내부 제품으로*.

**원칙 — 도구의 목적은 그 프로젝트에 가장 맞는 자동화를 설계하는 것이지, 특정 이력/JD 역량을 찍는 게 아니다.** (그 설계를 잘 하면 "agentic 제품" 역량이 증명되는 것은 *부수효과*일 뿐, 필드 선택의 동기가 아니다.) 이식성 최우선 — 특정 스택·언어·프레임워크에 종속되지 않는다.

- 사용자 응답 언어 = **한국어**, 브랜드 색 = violet.
- 산출은 **설계까지** — 에이전트 실행/오케스트레이션·스택별 스캐폴드 코드는 만들지 않는다.

## 인자

`/scout [대상 경로 또는 후보 slug (선택)]`:

- **인자 없음** → 현재 레포를 대상으로 **Phase A부터** 전체 실행.
- **대상 경로** → 그 경로를 스캔 대상으로 삼아 **Phase A부터** 실행.
- **후보 slug** → ledger(`.nova/scout-candidates.json`)에서 해당 canonical slug를 찾아 **Phase B~D(인터뷰·후보 생성·점수화)를 건너뛴다.** 단 Phase E 설계 스펙의 '컨텍스트' 필드를 채우려면 프로젝트 이해가 필요하므로 **Phase A(경량 프로젝트 이해)는 수행한 뒤 바로 Phase E(설계 스펙)** 로 간다. (ledger에 없으면 그 사실을 알리고 일반 실행으로 되돌린다.)

## 박제 패턴 — canonical "safe-action-agent" shape

`/scout`는 이 형태를 **후보 생성 렌즈이자 설계 골격**으로 쓴다:

1. 사람이 반복하는 **운영 판단** 1개를 대상으로.
2. 에이전트가 여러 시스템에서 **맥락 조사**(tool use) → **행동안 + 확신도/위험도** 제시.
3. **승인 게이트(HITL):** 저위험 자동 / 고위험 사람 승인.
4. 실행 + **자기검증**.
5. **eval + observability.**

> ⚠️ 이 5요소는 **상한(ceiling) 체크리스트지 전부 필수가 아니다.** 후보의 위험도·규모에 맞춰 **티어링**한다(§출력 b). 작은 read-only 자동화에 엔터프라이즈급 eval/obs를 강제하지 않는다 — 도구 자신의 YAGNI를 후보 산출 스펙에도 적용한다.

## 절차

### Phase A — 프로젝트 이해 (comprehension, *not* grep)

얕은 신호 스캔이 아니라 **프로젝트가 무엇을 하는지 이해**한다. 입력:

- README · docs · 매니페스트(package.json·pyproject 등)
- 도메인 모델(schema · types)
- surfaces · workflows(라우트 · 잡 · CLI · 외부연동)
- **git 히스토리 · 커밋** (`git log`, 파일 변경 빈도·재방문)

파일·구조 파악은 **Read/Glob/Grep**으로 한다 — `allowed-tools`는 일반 Bash를 의도적으로 제외(오직 `git`·`mkdir -p`만 Bash 허용)하므로 `cat/find/head/sed`를 Bash로 부르지 말 것.

큰 레포는 **Explore 서브에이전트(`Task`)로 위임**한다(메인 컨텍스트 보존, 깊이 적응형). Explore는 **고정 구조 반환 계약**으로 받아 Phase C에 먹인다:

```
{
  프로젝트 정체(도메인·사용자),
  도메인 엔티티,
  surfaces/workflows 목록,
  git 마찰 영역(파일·재방문빈도·메시지 신호),
  후보 seed 힌트
}
```

### Phase B — 짧은 인터뷰 (3문, tacit knowledge)

1. 사람이 **반복해서 손으로 판단/처리하는 일**은?
2. **틀리면 비싼**(오류비용) 결정은? *(→ S3 자동화 가치의 오류비용 성분; 빈도·수작업 성분은 Q1·Q3)*
3. 지금 **누가 어떤 도구로** 하나?

> **인터뷰는 사용자에게 실제로 물어본다.** comprehension-only fallback은 사용자가 응답할 수 없는 비대화형 상황이거나 무응답일 때만 사용 — 편의로 인터뷰를 생략하지 말 것.
>
> **Fallback:** 도메인을 아는 사람이 없어 인터뷰가 비면, **comprehension-only로 진행**하되 모든 후보를 *추정(unverified)* 으로 표기한다.

### Phase C — 후보 *생성* (박제 패턴을 generation lens로) + dedup

방법을 명문화한다(암묵 즉흥생성 금지):

1. **seed 나열:** 인터뷰 답 + git-마찰 영역 + surfaces/workflows를 후보 seed로 모은다.
2. **투영:** 각 seed를 박제 패턴 5요소 형태에 투영해 *"safe-action-agent로 성립하는가"* 판정. (기존 코드에 없어도 됨 — *있어야 할* 워크플로우도 seed가 될 수 있다.)
3. **등록:** 성립분만 후보로 등록.
4. **dedup:** 같은 운영 판단이 입도만 다르게 여러 후보로 잡히면, **행동 가능한 입도의 것 하나로 병합**(나머지는 sub-scope로 각주). 판단 밀도가 낮아 **이미 스크립트/룰로 처리되는** seed는 별도 후보로 올리지 말고(원칙: 판단없는반복=스크립트), 관련 상위 후보가 있으면 그 **각주(sub-scope)** 로 접는다.

> 박제 패턴 5요소 투영은 "**에이전트 형태가 되는가**"의 **coarse 스크린**(명백한 비-후보만 제거)이다. 정밀 게이트(G1~G3)와 점수는 **Phase D**가 한다 — C에서 애매하면 통과시켜 D가 판정.

### Phase D — 적합도 점수 → 랭킹 (정량)

각 후보를 §루브릭으로 **점수화**하고 **점수 근거와 함께** 랭킹한다(§출력 a).

> **0-후보 / HOLD-only 정직 반환:** 채점 가능한 후보(G2·G3 통과)가 하나도 없으면 **억지 Top-1을 만들지 말고** *"적합 후보 없음 + 이유(평가불가/무통제/수단없음)"* 를 반환한다. **보류(HOLD)만 남은 경우**는 "없음"이 아니라 *"eval 셋 확보 시 유망 — 보류 후보 N건"* 으로 구분해 반환한다.

### Phase E — 선택 후보 → 설계 스펙

Top-1(또는 사용자가 고른 후보)을 §출력 (b) 설계 스펙 문서로 출력한다.

> **스코프 확정 전 기존 구현 조사(필수).** comprehension(Phase A)은 후보 발굴용 이해지, 스코프 확정 근거가 아니다 — 그 수준에서 MVP를 확정하면 *이미 존재하는 것을 다시 짓는 스펙*이 나온다(실전 검증된 실패 모드). MVP 항목마다 실제 코드 경로를 열어 "이미 구현돼 있는가"를 파일 근거와 함께 확인하고, 이미 있는 항목은 스코프에서 제외하거나 명시적 재사용으로 표기한 뒤에만 스펙을 확정한다. grep 히트가 아니라 코드 경로 읽기 기준.

## 루브릭 (= "적합한 걸 찾기"의 알맹이)

**게이트 축 (pass/fail — 하나라도 fail이면 탈락 또는 '보류' 표기):**

- **G1 평가 가능** — 신뢰성 측정용 ground truth(과거 사례/정답)를 만들 수 있는가? *(전무 → 보류)*
- **G2 안전 통제** — 되돌릴 수 있거나 승인 게이트로 통제 가능한가? *(비가역·무통제 → 탈락)*
- **G3 행동 수단 존재 가능** — 최소한 상상 가능한 도구/API가 있는가? *(전무 → 탈락)*

> **보류(G1 fail) 처리:** G2·G3는 통과했는데 G1만 fail이면 **탈락이 아니라 보류** — 채점·랭킹에 포함하되 스코어카드에 `HOLD(ground-truth 부재)` 플래그를 달고, Top-1로 선택되면 **'eval 셋 확보'를 선결 리스크로 강제**한다.

**점수 축 (G2·G3 통과 후보 채점 · G1-fail은 HOLD로 채점 · 각 2/1/0 · 점수마다 이유 1줄 강제):**

| 축 | 2 | 1 | 0 |
|---|---|---|---|
| S1 반복성·빈도 | 자주 | 가끔 | 드묾 |
| S2 판단 밀도 | 재량·맥락 필요 | 일부 | 순수 기계작업(→스크립트) |
| S3 자동화 가치 (건당 절약 또는 회피 오류비용)×빈도 | 큼 | 중 | 미미 |
| S4 행동수단 성숙도 | 현존 API/시스템 | 구축 필요(+비용을 리스크로) | — (0은 G3서 탈락) |
| S5 입출력 명확성 | MVP로 떨어짐 | 다소 모호 | 경계 불명 |

> S3 척도 앵커: **2** = 자주(≥주간/시간 절약) 또는 큰 오류비용(고객·데이터·규정 영향) · **1** = 월간·분단위 절약 또는 중간 오류비용 · **0** = 드묾 + 경미

- **집계:** `sum(S1..S5)` 내림차순 랭킹.
- **동점 타이브레이크:** ① 자동화 가치(S3) → ② 가역성(되돌리기 쉬움) 정도.
- *양면성 각주:* '고위험·비가역'은 S3(가치)를 올리면서 G2/타이브레이크(안전)를 낮춘다 — 의도된 상반 신호.
- **원칙:** 판단 없는 반복 = 스크립트 · 평가 불가 = 보류 · 행동 수단 없음 = 탈락.

## 출력 (2종 + ledger)

### (a) 랭킹 스코어카드 (사용자에게 표시)

후보별 한 줄, 내림차순:

```
후보 slug | G1·G2·G3 pass/fail | S1..S5 점수 | 합계 | 한 줄 이유
```

### (b) 설계 스펙 문서 `docs/specs/<slug>-agent-design.md`

필드:

- **컨텍스트** (Phase A 이해 요약)
- **기존 구현 조사** (MVP 항목별 "현재 코드에 이미 있는가" — 파일 근거 포함; 있으면 제외/재사용 표기)
- **문제 · 대상 사용자**
- **에이전트 책임 vs 사람 책임** (HITL 경계)
- **도구 목록** (read · write · 위험도)
- **state · 플로우**
- **승인 게이트 규칙**
- **eval 셋** (아래 티어링)
- **observability 지표** (아래 티어링)
- **MVP 스코프 + 비-목표(YAGNI)**
- **미검증 가정 · 리스크**

**eval / obs 티어링 규칙:**

- *저위험 read-only/조회형:* 작은 golden set + 정확도/승인율. (regression · LLM-judge · prod feedback는 **N/A 명시** 또는 필요 시)
- *중위험 write · 가역:* + regression + LLM-judge + trace · cost · latency.
- *고위험/비가역 근접:* 전체 5요소 + human eval + production feedback loop.
- 박제 패턴 5요소는 상한 — 후보가 요구하지 않는 항목은 **"N/A + 이유"** 로 정직하게 비운다.

### (c) ledger `.nova/scout-candidates.json` (compounding)

랭킹 · 선택 · 기각 후보를 기록한다:

```json
[{ "slug": "...", "점수": 0, "gate판정": "...", "선택여부": false, "ts": "<실행시점>" }]
```

재실행 시 **이전 ledger 로드 → diff**("신규 후보 N / 점수 오른 후보 / 기각 유지")를 보여줘 프로젝트가 진화할수록 발굴이 복리가 되게 한다. (ts는 스킬 실행 시점 주입 — 날짜 단위, 모델 컨텍스트의 today's date 기준. 쉘 `date` 미사용, allowed-tools 제약.)

- **slug는 canonical 규칙**(대상 시스템 + 동작 동사 — 예: `orders-refund`, `onboarding-grant`)으로 생성한다 — LLM 재명명으로 같은 판단이 '가짜 신규 후보'로 뜨는 것을 막는다. diff는 canonical slug + 의미 매칭.

## 산출물 규칙

- **설계 스펙 경로는 고정** — `docs/specs/<slug>-agent-design.md`. 다른 경로로 흩뿌리지 않는다.
- **파일 생성만 · commit은 사용자에게** — Jay CLAUDE.md의 자동 commit 금지를 준수한다. `/scout`는 절대 스스로 `git add`/`git commit` 하지 않는다.
- 디렉토리 확보는 `mkdir -p`로 — `docs/specs/`, `.nova/` 를 쓰기 전에 보장한다.
- **ledger = `.nova/scout-candidates.json`** (머신 상태). `.nova/*.json`은 gitignore되어 커밋되지 않으며, `.nova/gate.on` 같은 opt-in 마커는 팀 공유를 위해 커밋 가능하게 남는다.
- 위치 = **`SKILL.md`만** (결정적 grep 스크립트 없음). 스캔/이해는 내장 검색도구 + Explore 서브에이전트로.
