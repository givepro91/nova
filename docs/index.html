왜 만들었나Why it exists

무거운 건 내려놓고,
정직한 하나만 다시 지었어요We let go of the heavy thing,
and rebuilt one honest one

예전 Nova는 훨씬 무거웠어요. Claude Code 위에 오케스트레이션과 품질 게이팅을 얹은 프레임워크였는데, 그 기능을 플랫폼이랑 주변 도구들이 따라잡아서 기본으로 해주기 시작했어요. 그러니 굳이 쓸 이유가 없어졌죠. 무게도 짐이 됐고요. 상태를 계속 갱신하느라 토큰을 먹고, 늘 켜져 있는 규칙들이 컨텍스트를 차지하고, 손도 많이 갔어요. 그래서 결국 안 쓰게 됐어요.

The old Nova was a lot heavier. It put orchestration and quality-gating on top of Claude Code, and then the platform and the tools around it caught up and started doing that by default. There wasn't much reason to keep using it. The weight became its own cost: updating state ate tokens, always-on rules took up context, and it needed constant upkeep. So it ended up unused.

여기서 하나 배웠어요. 빠르게 크는 플랫폼 위에 무거운 걸 올리면, 플랫폼이 그 기능을 흡수하는 순간 끝이에요. Nova가 못나서가 아니라 플랫폼이 좋아져서 생긴 일이고요. 딱히 숨길 일도 아니에요.

One lesson stuck. If you build something heavy on a fast-moving platform, the day it absorbs that feature, you're done. It wasn't that Nova was bad; the platform just got good. Nothing to hide there.

그래서 인정하고 거의 다 버렸어요. 진짜 쓸모 있던 하나만 들고 다시 만들었어요. So we admitted it and threw out almost everything. We kept the one part that actually pulled its weight and rebuilt around it.

그 하나는, 만드는 쪽과 검사하는 쪽을 따로 둬서 실제로 런타임 버그를 잡아낸 방식이었어요. 지금 Nova는 그걸 중심에 두고 있어요. That one part kept the maker and the checker separate, and it caught real runtime bugs. Nova is built around it now.

무엇인가What it is

에이전트를 대신 굴리진 않아요. 옆에서 챙겨요It doesn't run the agent. It looks after it

일을 굴리는 건 플랫폼이랑 주변 도구들이 이미 잘해요. Nova는 거기 안 끼어들어요. 대신 에이전트 옆에서 규칙을 지키고, 배운 걸 남기고, 끊긴 작업을 잇고, 한 일을 검증하고 기록해요. 플랫폼이 안 챙기는 부분이죠. 플러그인 하나에 loop 다섯 개예요. Running the work is something the platform and its tooling already do well. Nova stays out of that. Instead it sits next to the agent and keeps the rules, saves what you taught it, carries unfinished work forward, and checks and records what got done. That's the part the platform doesn't cover. One plugin, five loops.

규칙Rules/claude-md

CLAUDE.md에 작업 규율과 검증 방식, git 워크플로를 담는 관리 블록이에요. 몇 번을 다시 써도 한 벌만 깔끔하게 남아요.A managed block in CLAUDE.md that holds your working rules, how to verify, and the git workflow. Re-run it any number of times and it stays one clean copy.

학습Learning/learn

한 번 고쳐준 걸 규칙 하나로 남겨요. 같은 실수를 다음 세션이 반복하지 않게요.Turns one correction into one lasting rule, so the next session doesn't repeat the same mistake.

연속성Continuity/handoff

브랜치마다 핸드오프 메모를 git에 남겨요(SessionStart·PreCompact·Stop 훅). 세션이 끊겨도 다음이 그대로 이어받아요.Leaves a per-branch handoff note in git (SessionStart, PreCompact, Stop hooks). A fresh session picks up exactly where you stopped.

검증Verification/gate

완료라고 말한 걸 독립 검증자가 다시 봐요. 진짜 했는지, 테스트는 돌렸는지, 시킨 것만 했는지 증거로 확인해요.An independent verifier re-checks what you called done. Did it really happen, did the tests actually run, did you stick to what was asked — against the evidence.

기록Record/worklog

세션을 정리한 worklog를 남겨요. 원하면 Nova 톤의 단독 HTML로도 뽑고요. 한 일이 그냥 사라지지 않아요.Leaves a worklog that sums up the session, and if you want, a standalone HTML in Nova's style. What you did doesn't just vanish.

핵심The core

"다 됐다"고 할 때, 진짜일까When the agent says "done," is it really?

에이전트가 적당히 건너뛰고 "다 했다"고 말할 때가 있어요. 맡기는 일이 많아질수록 이게 더 문제가 되고요. 그런데 플랫폼이 자기 에이전트의 보고를 의심하는 기능을 직접 넣어줄 리는 없어요. 그 자리를 Nova가 맡습니다. Sometimes an agent skips a step and still reports it as done. The more you hand off to it, the more this matters. And a platform isn't going to ship a feature that doubts its own agent's word. Nova takes that job.

?테스트, 진짜 돌렸어요?Did you really run the tests? 말만 한 건지, 출력이 있는지.— a claim, or actual output.
?시킨 범위는 지켰어요?Did you stay in scope? 요청 밖을 건드린 건 없는지.— or touch what wasn't asked.
?검증 안 한 걸 done으로 포장했어요?Was the unverified dressed up as done? 증거 없는 완료 선언인지.— a completion claim with no evidence.

데모Demo

claim → evidence, 실제 한 번claim → evidence, for real

검증자를 따로 띄워서, 세션이 한 말을 실제 git diff랑 파일, 테스트 결과에 맞춰봐요. 코드 diff만 봐서는 안 보이는 게 여기서 걸려요.It spins up a separate verifier that checks what the session claimed against the real git diff, the files, and the test output. Things a diff-only review misses show up here.

/gate · independent verifier
> /gate
# 세션의 완료 주장을 증거에 대조 중…auditing the session's completion claims against evidence…

claim 1  "로그인 버그 고침fixed the login bug"          ✓ 증거: auth.ts:42 diff 일치evidence: auth.ts:42 diff matches
claim 2  "테스트 통과tests pass"               ✗ 증거 없음 — 실행 출력이 transcript에 없음no evidence — no run output in transcript
claim 3  "그 파일만 건드림only touched that file"        ✗ 범위 이탈 — config.ts 도 수정됨scope drift — config.ts also modified

verdict  NOT DONE · 3건 중 2건 미검증. critical → 2라운드 재감사.2 of 3 unverified. critical → re-audit, round 2.

예시 출력이에요. 실제 /gate는 세션 안에서 전체 맥락을 보고 진짜 git diff와 파일, 테스트 결과를 읽어요. Sample output. The real /gate runs inside the session with full context, reading the actual git diff, files, and test results.

왜 좋은가Why it's good

네 가지 이유Four reasons

01 매번 리셋되지 않아요Nothing resets

이번 세션에서 고친 것, 못 끝낸 것, 검증한 것, 한 일이 다음 세션으로 넘어가요. 매번 처음부터 다시 하지 않아요.What you fixed, what you didn't finish, what you verified, what you did — it all carries into the next session. You don't start over each time.

02 흡수 안 되는 정직성 게이트An honesty gate they can't copy

맡기는 일이 많아질수록 쓸모가 커져요. 그리고 플랫폼은 자기 에이전트를 의심하는 기능을 스스로 넣기 어렵고요.It gets more useful the more you hand off. And a platform isn't well placed to build something that doubts its own agent.

03 가볍고, 거의 공짜Lean, almost free

플러그인 하나에 훅은 전부 직접 켜는 방식이에요. 안 켜면 아무것도 안 해요.One plugin, and every hook is something you switch on yourself. Leave it off and it does nothing.

04 스스로 점검해요It checks itself

만들면서 codex와 공식 문서, 코드리뷰로 검증받았어요. 그리고 Nova 자신의 /gate로 스스로를 점검해요.It was checked against codex, the official docs, and code review while being built. And it runs its own /gate on itself.

FAQ

자주 묻는 질문Frequently asked

이게 예전 "AI Agent Ops Framework" Nova인가요?Is this the old "AI Agent Ops Framework" Nova?
아니요. 이름만 같지 사실상 다른 물건이에요. 그건 무겁고 큰 프레임워크였고, 이건 가벼운 플러그인 하나입니다. 레포도 방향도 다릅니다.No. Same name, but really a different thing. That was a big, heavy framework; this is one lightweight plugin. Different repo, different direction.
느려지지 않나요? 컨텍스트 세금은요?Will it slow me down? What about context-tax?
훅을 직접 켜기 전까지는 아무것도 안 돌아가요. 커맨드도 필요할 때만 부르면 됩니다. 늘 켜져서 무거워지는 일이 없어요.Nothing runs until you switch a hook on yourself. Commands only fire when you call them. There's nothing sitting in the background weighing things down.
멀티 에이전트 오케스트레이션도 하나요?Does it do multi-agent orchestration?
안 합니다. 그건 플랫폼이랑 omc가 하는 일이에요. Nova는 일부러 거기 안 들어갑니다. 에이전트가 일하는 동안 옆에서 규칙이랑 기록, 검증을 챙기는 쪽이에요.It doesn't. That's what the platform and omc are for. Nova deliberately stays out of it, and works alongside the agent instead — keeping the rules, the records, and the checks.
회사 내부 도구나 서버가 필요한가요?Does it need internal tools or a server?
아니요. 회사 내부 도구 같은 건 하나도 안 씁니다. git만 있으면 되고, 뽑아내는 결과물(worklog HTML 등)도 외부 의존 없이 그 파일 하나로 열려요.No. It uses none of that. All it needs is git, and the things it produces (like the worklog HTML) open as a single file with nothing external to load.
계속 업데이트되나요?Is it still updated?
네, 대신 가볍게요. 플랫폼이 바뀌거나 고칠 게 생겼을 때만 /nova-dev로 손봐요. 스캐너나 cron으로 자동으로 막 돌리는 건 없앴어요.Yes, but lean. A single one-sitting loop, /nova-dev, updates it only when the platform shifts or there's a concrete improvement. No heavy auto-evolution (scanners, cron).
